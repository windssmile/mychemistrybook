## 3. 单元生产循环

每个单元依次完成：

1. 核对蓝图中的中心问题、边界和学习成果；
2. 核对本单元只依赖此前核心单元或独立补给站；
3. 设计引入情境和概念主线；
4. 按8.1的页面结构写核心叙事；
5. 配置概念辨析、标准推理或计算、迁移应用三类示例；
6. 配置即时检查、常见误区、知识结构和课后练习；
7. 提供提示、思路和完整答案；
8. 补齐下一课衔接、术语、数据条件和资料来源；
9. 检查Markdown、MathJax、mhchem、折叠块和站内链接；
10. 运行`mkdocs build --strict`。

单元初稿不因字数达标而完成。验收依据是学习成果是否被正文和练习覆盖，
关键推理是否完整，以及学习者是否能判断结论的条件与边界。

## 4. 章内完成后的整体复核

一章全部单元完成后，进行一次章内闭环：

1. **范围复核**：有无越界讲授后续章节内容，有无遗漏蓝图成果；
2. **依赖复核**：前置链接是否真实可用，是否出现循环依赖；
3. **叙事复核**：各单元中心问题是否连续，是否重复建立同一概念；
4. **术语复核**：符号、单位、中文译名和模型边界是否一致；
5. **练习复核**：是否覆盖识别、解释、计算、图表和迁移；
6. **小结复核**：章末小结是否准确反映全章，而非逐节标题拼接；
7. **技术复核**：导航、公式、化学式、折叠答案和移动端长公式是否正常；
8. **来源复核**：术语、数据、图表和外部事实是否记录权威来源。

通过后再把章状态改为“核心初稿完成”。

## 5. 跨章回看机制

为避免局部正确、全书冲突，设置三类回看：

- **相邻章回看**：完成第\(n\)章后，复核第\(n-1\)章的“下一章”衔接，
  并核对第\(n+1\)章蓝图中的前置条件；
- **理论—应用回看**：元素章节调用原理章节时，如果发现原理表述不足，
  优先在原主讲章节补充解释或链接，不在元素章节重新建立一套理论；
- **阶段回看**：第5、8、11、16、18章完成后，对已完成路线做知识归属、
  符号和综合练习复核。

任何跨章修改都应保留“一个知识点只有一个主讲位置”的原则。

## 6. 状态约定

页面使用以下状态：

- `planned`：蓝图已确定，尚未撰写；
- `draft-v0.1`：结构完整的首轮自学稿；
- `chapter-reviewed`：已通过章内整体复核；
- `cross-reviewed`：已在相关后续章节完成后回看；
- `validated`：已完成科学审校、技术检查和目标读者盲学测试。

当前按`draft-v0.1 → chapter-reviewed → cross-reviewed`推进；8.1已在第8章完整上下文
中完成章内及后续沉淀平衡方向的跨章复核。
