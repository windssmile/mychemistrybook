# 开发与编写踩坑记录

> 用途：记录课程设计、内容编写、渲染、部署和协作中已经遇到的问题，避免后续重复踩坑。
> 维护原则：记录“现象—原因—解决—预防”，不只记录最终命令。
> 最近更新：2026-07-23

## 1. 课程设计与内容依赖

### 1.1 Born–Haber循环不能早于Hess定律

**现象**

最初把晶格能和Born–Haber循环都安排在第5章“晶体与固体基础”，但Hess定律要到第6章才学习。

**原因**

按知识主题归类时，只看到Born–Haber循环用于晶格能，忽略了它在方法上依赖热化学状态函数和Hess定律。

**解决**

- 第5章只定性建立晶格能及电荷、半径影响；
- 第6.3单元学习Hess定律后，再定量建立Born–Haber循环。

**预防**

安排知识点时同时检查：

1. 它属于哪个主题；
2. 它实际使用了哪些方法；
3. 这些方法是否已经学过。

### 1.2 沉淀章不能提前使用配位形成常数

**现象**

最初计划在第9章定量处理“沉淀—配位耦合”，但配位形成常数到第11章才正式建立。

**原因**

希望展示多平衡耦合，却造成了隐性循环依赖。

**解决**

- 第9章只组合酸碱、沉淀和题目已给出的副反应；
- 配体促进溶解只作定性预览；
- 形成常数和定量配位平衡留到第11章。

**预防**

每章“必须前置”只能指向此前核心单元。综合性强不等于可以提前使用未定义的工具。

### 1.3 普通命名不能偷偷提前系统讲氧化数

**现象**

第1.1单元的早期版本要求学习者在名称、化学式和氧化数之间转换，但氧化数规则的主讲位置是第10.1单元。

**原因**

无机物命名常用氧化态信息，容易无意中把后续概念提前。

**解决**

- 第1.1只使用常见离子、电荷和基础命名；
- 氧化数的系统规则、判断和氧化还原意义归第10.1。

**预防**

区分“为完成当前任务提供少量约定”和“系统建立一个概念”。

### 1.4 d区元素不全部等同于严格定义的过渡元素

**现象**

早期标题把Zn、Cd、Hg不加说明地归入“过渡元素”。

**原因**

国内教材常把d区和过渡元素并列或合并处理，但严格定义需要考虑原子或常见阳离子是否具有未充满d亚层。

**解决**

- 在第16.6明确讨论“d区”与“过渡元素”的定义差异；
- 标题改为“Zn、Cd、Hg与d区反应网络”。

**预防**

遇到教材习惯用语和严格术语可能不同的地方，必须设置术语说明。

### 1.5 详细不等于无限扩张

**现象**

第7、14、16章容易分别变成动力学/平衡公式清单、主族物质百科和逐元素性质清单。

**原因**

每章承担内容多，自学教材又要求解释充分，容易通过增加内容解决焦虑。

**解决**

- 固定每单元一个中心问题；
- 元素化学只选能体现趋势、结构—性质或重要反应网络的体系；
- 提高和进阶内容独立编号；
- 用盲学时间而不是字数判断密度。

**预防**

新增内容前先回答：它是否直接支持本单元学习成果？如果不是，移入资料卡或进阶路线。

## 2. Markdown、公式与内容渲染

### 2.1 `<details>`内部默认不会解析Markdown

**现象**

折叠答案中的表格、列表和`\ce{}`在构建后的HTML里显示为原始文本。

**原因**

原生HTML块中的内容不会自动按Markdown继续解析。仅启用`md_in_html`扩展还不够。

**解决**

把：

```html
<details>
```

改为：

```html
<details markdown="1">
```

并保留`md_in_html`扩展。

**预防**

- 内容模板从一开始就使用`<details markdown="1">`；
- 构建后搜索折叠块内是否仍存在未包装的`\ce{}`；
- 不只检查源Markdown，要检查生成HTML。

### 2.2 列表中的显示公式可能失去数学分隔符

**现象**

有序列表中的：

```markdown
3. 在反应

   \[
   \ce{HCl + H2O -> H3O+ + Cl-}
   \]
```

构建后变成普通的方括号文本，没有进入MathJax。

**原因**

显示公式位于列表缩进上下文中，Markdown解析器没有把`\[...\]`识别为独立数学块。

**解决**

短方程改为行内数学：

```markdown
3. 在反应\(\ce{HCl + H2O -> H3O+ + Cl-}\)中……
```

如果必须显示为块，应使用经过实际构建验证的列表内块级写法。

**预防**

- 列表题干优先使用行内公式；
- 构建后检查是否出现`<p>[ ... ]</p>`；
- 不根据Markdown源文件“看起来正确”判断最终渲染。

### 2.3 MathJax不会自动支持`\ce{}`

**现象**

普通LaTeX可以显示，但`\ce{NH3}`等化学式无法解析。

**原因**

`\ce{}`来自mhchem扩展，不是MathJax核心命令。

**解决**

在MathJax配置中显式加载：

```javascript
loader: {
  load: ["[tex]/mhchem"]
},
tex: {
  packages: {
    "[+]": ["mhchem"]
  }
}
```

**预防**

- 首页保留一个化学方程式渲染测试；
- 本地构建后确认页面加载`mathjax.js`和MathJax主脚本；
- 浏览器验证`\ce{}`是否已转成MathJax输出，而不是只看HTML中存在脚本标签。

### 2.4 即时导航需要重新触发MathJax排版

**现象**

Material主题启用`navigation.instant`后，页面切换不一定发生完整浏览器刷新。

**原因**

MathJax默认只在初次页面加载时排版。

**解决**

订阅Material提供的`document$`，在每次即时导航后执行：

```javascript
MathJax.startup.output.clearCache();
MathJax.typesetClear();
MathJax.texReset();
MathJax.typesetPromise();
```

**预防**

浏览器测试必须包含“从首页点击进入教材页”，不能只直接打开深层URL。

### 2.5 目录链接最好指向实际Markdown文件

**现象**

首页链接`chapters/08-acid-base/`可以在最终网站中工作，但`mkdocs build --strict`提示它不是可识别的源文件链接。

**原因**

MkDocs在构建时按源文件解析内部链接，不把目录URL自动映射到`index.md`。

**解决**

源Markdown写：

```markdown
[开始阅读](chapters/08-acid-base/index.md)
```

MkDocs会在构建时转换为目录URL。

**预防**

内部链接一律指向`.md`源文件，并始终使用`mkdocs build --strict`。

### 2.6 标题中的公式会污染自动目录

**现象**

正文中的化学式已经由MathJax正确渲染，但含有`\(\ce{OH-}\)`的标题在页面目录和可访问性快照中显示为原始TeX标记。

**原因**

标题正文和MkDocs生成目录所使用的纯文本不是同一条渲染路径；MathJax可以处理文章区域，却不会可靠地重写所有导航标签、锚点和目录文本。

**解决**

- 标题使用“氢氧根”“水合氢离子”等可读中文；
- 公式放在标题后的正文首句；
- 不让数学或化学公式承担标题的唯一语义。

**预防**

- 标题和导航标签默认不用LaTeX或mhchem；
- 浏览器验收同时检查正文、左侧导航、右侧目录和页面标题；
- 如确实需要符号，使用普通Unicode且确认锚点稳定。

## 3. MkDocs与本地构建

### 3.1 `docs/`和发布内容目录不能混用

**现象**

项目已经用`docs/`保存课程基线和制作规范，而MkDocs默认也把`docs/`作为网站内容目录。

**原因**

规划文档和学生可见内容的用途不同。

**解决**

- 网站源内容放在`content/`；
- 课程设计和内部规范保留在`docs/`；
- `mkdocs.yml`设置`docs_dir: content`。

**预防**

项目早期就区分“发布内容”和“项目文档”。

### 3.2 GitHub Pages项目站点不是域名根路径

**现象**

仓库站点地址是：

```text
https://windssmile.github.io/mychemistrybook/
```

而不是：

```text
https://windssmile.github.io/
```

**原因**

普通仓库发布的是GitHub Pages项目站点，带有仓库名路径。

**解决**

- `site_url`通过`SITE_URL`环境变量设置；
- GitHub Actions构建时使用仓库所有者和仓库名生成URL；
- 本地预览使用`http://127.0.0.1:8000/`。

**预防**

不要在模板、CSS或正文中写以`/`开头的站点根绝对路径。

### 3.3 Material for MkDocs会输出MkDocs 2.0迁移警告

**现象**

使用Material 9.7.7构建时，会输出关于未来MkDocs 2.0不兼容的醒目警告。

**原因**

这是Material项目主动输出的生态迁移提示，不是当前构建失败。

**解决**

- 当前固定`mkdocs==1.6.1`和`mkdocs-material==9.7.7`；
- 只在独立升级任务中更改主版本；
- 构建成功仍以退出码和实际HTML为准。

**预防**

固定依赖版本，不使用无上限的`pip install mkdocs-material`作为生产构建。

### 3.4 沙箱中可能无法绑定本地预览端口

**现象**

`mkdocs build`成功，但`mkdocs serve --dev-addr 127.0.0.1:8000`报：

```text
PermissionError: [Errno 1] Operation not permitted
```

**原因**

当前执行沙箱禁止进程绑定本地网络端口。这与MkDocs配置是否正确无关。

**解决**

- 使用允许端口绑定的执行权限启动预览服务；
- 如果只验证构建，可直接检查`site/`产物；
- 浏览器视觉验证需要服务成功运行。

**预防**

把“静态构建验证”和“本地服务验证”作为两个独立检查步骤。

### 3.5 沙箱网络不能直接从PyPI安装依赖

**现象**

首次运行`pip install -r requirements.txt`时出现DNS/连接失败，最终显示找不到任何版本。

**原因**

不是依赖版本不存在，而是沙箱默认限制网络访问。

**解决**

在明确许可后使用允许网络的安装步骤；安装固定版本：

```text
mkdocs==1.6.1
mkdocs-material==9.7.7
```

**预防**

看到“from versions: none”时同时检查网络错误，不能立即判断包名或版本写错。

## 4. GitHub与GitHub Pages

### 4.1 开始时本地目录不是Git仓库

**现象**

`git status`返回“not a git repository”。

**原因**

工作区最初是空目录，没有`.git`和远程地址。

**解决**

先完成本地内容和构建；目标仓库明确后再初始化并连接：

```text
https://github.com/windssmile/mychemistrybook.git
```

**预防**

发布前必须确认本地Git状态、默认分支和远程目标，不能假设工作区已经克隆。

### 4.2 GitHub CLI存在，但登录令牌已失效

**现象**

`gh --version`正常，但`gh auth status`显示`windssmile`令牌无效。

**原因**

安装CLI不等于当前会话仍有有效认证。

**解决**

推送前重新认证：

```bash
gh auth login -h github.com
```

认证后再次运行：

```bash
gh auth status
```

**预防**

任何GitHub写入流程开始前都同时检查CLI存在和认证状态。

### 4.3 必须先明确目标仓库

**现象**

本地没有远程地址，不能安全决定推送目标或Pages URL。

**原因**

GitHub发布是外部写操作；仓库所有者、名称、可见性和默认分支都会影响结果。

**解决**

目标仓库明确为：

```text
windssmile/mychemistrybook
```

该仓库为公开仓库，默认分支为`main`。

**预防**

不要凭本地目录名猜远程仓库。

### 4.4 非Jekyll构建应使用GitHub Actions

**现象**

MkDocs需要先把Markdown构建成静态HTML，不能只把`content/`作为Pages目录直接发布。

**原因**

GitHub Pages的分支发布不会自动运行MkDocs。

**解决**

使用自定义Actions工作流：

1. checkout；
2. setup-python；
3. 安装依赖；
4. `mkdocs build --strict`；
5. `upload-pages-artifact`；
6. `deploy-pages`。

**预防**

仓库设置中Pages来源必须选择`GitHub Actions`，不是`Deploy from a branch`。

### 4.5 GitHub Actions版本会变化

**现象**

较旧示例仍使用`actions/checkout@v4`、`upload-pages-artifact@v3`等版本。

**原因**

Actions运行时和Node版本会更新，网上教程可能过时。

**解决**

搭建时依据GitHub当前官方文档使用：

- `actions/checkout@v6`；
- `actions/setup-python@v6`；
- `actions/configure-pages@v5`；
- `actions/upload-pages-artifact@v4`；
- `actions/deploy-pages@v4`。

**预防**

新增或升级工作流时查官方GitHub文档和官方Action仓库，不直接复制旧博客。

## 5. 每次发布前的最小检查

```text
[ ] 章节依赖没有指向后续未学概念
[ ] mkdocs build --strict 成功
[ ] 没有无法识别的内部链接
[ ] `<details>`内部公式和表格已实际渲染
[ ] 列表中的公式没有变成普通方括号文本
[ ] 首页和深层页面都能显示`\ce{}`
[ ] 即时导航后MathJax仍工作
[ ] 移动端导航和长公式可用
[ ] GitHub CLI认证有效
[ ] 远程仓库和默认分支正确
[ ] Pages来源设置为GitHub Actions
[ ] Actions构建和deploy任务成功
[ ] 线上URL和资源路径正确
```
